# 数据库 ER 图详细解说

本文档对项目中所有表的设计思路、字段含义、以及表与表之间的关系做逐项说明，便于理解 ER 图背后的业务与建模逻辑。

---

## 一、系统与 ER 设计目标简述

本系统是一个 **非遗（非物质文化遗产）展示与互动平台**，主要业务包括：

- 展示丽水市非遗项目、传承人、传承故事、资讯、活动等 **内容**；
- 用户 **注册 / 登录**，对项目进行 **收藏**、**浏览** 等行为，对活动进行 **报名**；
- 根据用户行为（浏览、收藏）和 **推荐权重配置** 做个性化推荐。

ER 图要表达的就是：**有哪些实体（表）**、**实体有哪些属性（字段）**、**实体之间是什么关系（1:1、1:N、M:N）**。关系用主键、外键或“逻辑关联”体现。

---

## 二、各表详细解说

### 1. sys_user（用户表）

| 项目 | 说明 |
|------|------|
| **表名** | sys_user |
| **用途** | 存储所有使用系统的人：既有普通用户，也有管理员，通过 `role` 区分。 |
| **主键** | id（自增） |
| **唯一约束** | username、email、phone 均唯一，保证登录与联系不重复。 |

**字段说明：**

- **username**：登录账号，字母开头、仅字母数字，4～10 位，用于登录。
- **password**：加密后的密码，不明文存储。
- **nickname**：昵称，前台展示用，可与账号不同。
- **avatar**：头像图片 URL，可为空。
- **email / phone**：邮箱、手机，用于联系或找回账号，唯一。
- **role**：枚举，USER（普通用户）或 ADMIN（管理员），决定权限。
- **create_time**：注册时间，由系统在插入时自动写入。

**在 ER 中的位置**：用户是“谁”的主体，与收藏表、行为表、报名表都是一对多（一个用户多条记录），因此用户表是多个关系的“一”端。

---

### 2. ich_project（非遗项目表）

| 项目 | 说明 |
|------|------|
| **表名** | ich_project |
| **用途** | 存储丽水市各类非遗项目的基本信息，是平台的核心展示内容之一。 |
| **主键** | id（自增） |

**字段说明：**

- **title**：项目名称，必填。
- **category**：分类，按丽水市所辖区县（如莲都区、龙泉市、青田县）等划分。
- **ich_type**：非遗类型，如民间文学、传统音乐、传统舞蹈等；与推荐权重表按类型对应。
- **description**：项目描述/介绍，长文本或富文本。
- **images**：多张图片 URL，在库中常用逗号分隔存储。
- **approval_time**：批准时间（列入非遗的时间）。
- **heritage_level**：遗产级别，如国家级、省级、市级、县级。

**在 ER 中的位置**：项目是“被收藏、被浏览”的对象，与收藏表、行为表是多对一（多条记录指向一个项目）；与权重表通过 `ich_type` 做逻辑上的“按类型”关联，无外键。

---

### 3. user_favorite_project（用户收藏项目表）

| 项目 | 说明 |
|------|------|
| **表名** | user_favorite_project |
| **用途** | 记录“哪个用户收藏了哪个非遗项目”，是 **用户 与 项目 多对多关系的中间表**。 |
| **主键** | id（自增） |

**字段说明：**

- **user_id**：用户 id，外键指向 sys_user.id，表示“谁”收藏。
- **project_id**：非遗项目 id，外键指向 ich_project.id，表示收藏了“哪个项目”。
- **create_time**：收藏时间，插入时自动写入。

**设计要点：**

- 若没有这张表，用户与项目就是多对多，无法在关系型数据库中直接表达（不能在一个表里存“多对多”）。
- 引入中间表后：  
  - 用户 → 收藏表：一个用户可有多条收藏记录 → **1 : N**；  
  - 收藏表 → 项目：每条记录只对应一个项目，一个项目可被多条记录引用 → **N : 1**。  
- 合起来即：**用户 M : N 项目**。业务上可加唯一约束 (user_id, project_id)，保证同一用户对同一项目只收藏一次。

**在 ER 中的位置**：夹在用户表和项目表之间，左右各一条线：用户 1:N 收藏表，收藏表 N:1 项目。

---

### 4. user_behavior（用户行为表）

| 项目 | 说明 |
|------|------|
| **表名** | user_behavior |
| **用途** | 记录用户的各种行为（浏览项目、浏览传承人、浏览故事、浏览活动、报名活动等），用于统计与推荐。 |
| **主键** | id（自增） |

**字段说明：**

- **user_id**：用户 id，可空——未登录用户也可能产生行为（如浏览），用空表示。
- **type**：行为类型，如 VIEW_PROJECT、VIEW_INHERITOR、VIEW_STORY、VIEW_ACTIVITY、REGISTER_ACTIVITY 等。
- **project_id**：关联的非遗项目 id，可空——只有与项目相关的行为才填，其他类型可为空。
- **create_time**：行为发生时间。

**设计要点：**

- 从“用户–项目”的角度看：一个用户可对多个项目产生行为，一个项目可被多个用户行为 → **用户 与 项目 通过行为也是多对多**。
- 因此行为表在这里也充当“用户–项目”多对多的 **中间表**：  
  - 用户 → 行为表：1 : N；  
  - 行为表 → 项目：N : 1（仅针对有 project_id 的记录）。  
- 与收藏表不同：行为表一条记录表示“一次行为”，同一用户对同一项目可以有多条浏览记录（不唯一约束），用于做浏览热度、推荐权重等。

**在 ER 中的位置**：同样夹在用户表与项目表之间，表示“用户 M : N 项目（通过行为）”。

---

### 5. ich_activity（非遗活动表）

| 项目 | 说明 |
|------|------|
| **表名** | ich_activity |
| **用途** | 存储非遗相关活动信息，用户可查看并报名参与。 |
| **主键** | id（自增） |

**字段说明：**

- **title**：活动标题。
- **region**：所属区县，与项目、资讯等按地区归类一致。
- **content**：活动内容介绍，长文本。
- **location**：举办地点。
- **start_time / end_time**：报名开始、报名结束时间。
- **cover_image**：封面图 URL。
- **create_time**：活动创建/发布时间。

**在 ER 中的位置**：活动是独立内容表，与用户没有直接外键；和用户的关系通过“报名表”建立，见下。

---

### 6. activity_registration（活动报名表）

| 项目 | 说明 |
|------|------|
| **表名** | activity_registration |
| **用途** | 记录“哪个用户报名了哪场活动”，是 **用户 与 活动 多对多关系的中间表**。 |
| **主键** | id（自增） |
| **唯一约束** | (user_id, activity_id)，保证同一用户对同一活动只报名一次。 |

**字段说明：**

- **user_id**：用户 id，外键指向 sys_user.id。
- **activity_id**：活动 id，外键指向 ich_activity.id。
- **register_time**：报名时间，插入时自动写入。

**设计要点：**

- 用户与活动是多对多：一个用户可报多个活动，一个活动可被多个用户报名。
- 中间表拆成：用户 1:N 报名表 N:1 活动；唯一约束避免重复报名。

**在 ER 中的位置**：用户表 —(1:N)— 报名表 —(N:1)— 活动表，表示用户 M : N 活动。

---

### 7. recommendation_weight（推荐权重表）

| 项目 | 说明 |
|------|------|
| **表名** | recommendation_weight |
| **用途** | 按“非遗类型”配置推荐算法的权重：基础权重、浏览增量、收藏增量等。 |
| **主键** | id（自增） |
| **唯一约束** | ich_type，一种类型一条配置。 |

**字段说明：**

- **ich_type**：非遗类型（与 ich_project.ich_type 取值一致），如“传统音乐”“传统舞蹈”。
- **base_weight**：该类型的基础权重。
- **view_weight**：用户每浏览一次该类型项目，增加的权重。
- **favorite_weight**：用户每收藏一次该类型项目，增加的权重。

**设计要点：**

- 表中 **没有 project_id**，不会直接引用具体项目，因此与项目表 **没有主外键关系**。
- 关联方式：通过 **ich_type** 在业务上对应——  
  - 一条权重记录对应一种类型；  
  - 多个非遗项目可以同属一种类型（ich_project.ich_type）。  
- 所以是 **权重表 1 : N 项目表**（按类型：一种类型一条配置，多条项目属于该类型）。在 ER 图中可画成“按 ich_type 逻辑关联”或虚线连接并标注“按类型”。

**在 ER 中的位置**：与 ich_project 之间为“按类型”的 1:N，不画传统外键线，可标注“逻辑关联”或“ich_type”。

## 三、关系总览与画图要点

### 1. 多对多为什么用中间表？

关系型数据库中，表与表之间只能直接表达“一对多”（通过外键）。  
**多对多**必须拆成两个“一对多”，中间引入一张表，这张表通常叫 **关联表 / 中间表**：

- 中间表每条记录表示“一次关联”（如一次收藏、一次报名）。
- 左边实体 —(1:N)— 中间表 —(N:1)— 右边实体 ⇒ 左边 M : N 右边。

本系统中：

- **用户 M : N 项目（收藏）**：中间表 = user_favorite_project。
- **用户 M : N 项目（行为）**：中间表 = user_behavior（仅看与 project_id 相关的记录）。
- **用户 M : N 活动**：中间表 = activity_registration。

画 ER 时：先画用户、项目、活动等实体，再画三张中间表，然后从用户到中间表画 1:N，从中间表到项目/活动画 N:1，在连线旁标注 1、N 或 M:N 即可。

---

### 2. 权重表与项目表如何表达？

- **没有外键**：recommendation_weight 不存 project_id，ich_project 也不存 weight_id。
- **逻辑关联**：通过 **ich_type** 在业务上对应——权重表按类型配置，项目表按类型归属。
- **基数**：一种类型一条权重，多个项目可属同一类型 → 权重表 1 : N 项目表（按类型）。

在 ER 图中可以：

- 画一条从 recommendation_weight 到 ich_project 的线，旁边标注“按 ich_type”或“逻辑关联”；或  
- 在说明文字中写明“通过 ich_type 对应，1 类型 : N 项目”。

---

### 3. 独立内容表怎么画？

ich_news、ich_inheritor、ich_inheritance_story、ich_activity 与用户、项目都没有主外键，在 ER 中：

- 各画一个实体矩形，列出主要属性即可。
- 不画指向用户表或项目表的外键线。
- 活动表与用户的关系通过 activity_registration 体现，所以活动表只需和报名表有一条 N:1 的线。

## 四、ER 思路小结（便于记忆）

| 关系 | 说明 | 在 ER 中的画法 |
|------|------|----------------|
| 用户 ↔ 项目（收藏） | 多对多，中间表 user_favorite_project | 用户 1:N 收藏表 N:1 项目 |
| 用户 ↔ 项目（行为） | 多对多，中间表 user_behavior | 用户 1:N 行为表 N:1 项目 |
| 用户 ↔ 活动 | 多对多，中间表 activity_registration | 用户 1:N 报名表 N:1 活动 |
| 权重表 ↔ 项目表 | 按 ich_type 逻辑 1:N，无外键 | 标注“按 ich_type”或逻辑关联线 |
| 资讯 / 传承人 / 传承故事 / 活动 | 独立内容，无指向用户/项目的外键 | 独立实体，活动再连报名表 |

按“先实体、再中间表、再连线、最后标权重表与项目的逻辑关系”的顺序画，就能得到与当前设计一致的 ER 图；需要更细的字段说明时，可回头查阅本文“二、各表详细解说”部分。
