# 第一部分 项目与业务

## 一、项目定位与目标

### 1.1 一句话概括

本系统是一个基于 **Vue.js 前端 + Spring Boot 后端 + MySQL 数据库** 的**丽水市非物质文化遗产数字化展示与管理平台**，面向两类用户（普通用户、管理员），实现非遗资源的集中展示、活动预约、用户收藏与行为记录、基于类型权重的个性化推荐，以及后台的内容管理与数据统计。

### 1.2 业务背景（答辩时可简述）

丽水市拥有丰富的非遗资源（如龙泉青瓷、畲族民歌等），传统传播方式受地域、时间限制。本平台通过数字化手段整合本地非遗信息，以 Web 形式向公众展示非遗项目、传承人、传承故事、资讯与活动，并提供活动预约、收藏、AI 咨询等功能；管理员可在后台对各类内容进行增删改查，并配置推荐权重、查看统计报表，从而提升丽水非遗的社会知晓度与传播效率。

### 1.3 用户角色与权限

| 角色       | 说明与权限 |
|------------|------------|
| **普通用户** | 可注册、登录；浏览非遗项目、资讯、传承人、传承故事、活动（部分接口无需登录）；登录后可收藏项目、预约活动、使用 AI 咨询、修改个人资料；首页推荐会根据其浏览与收藏行为进行个性化展示。 |
| **管理员**   | 使用独立登录入口（/login_admin），仅 role=ADMIN 的用户可登录；登录后访问后台（/admin 下所有页面），对非遗项目、资讯、传承人、传承故事、活动、用户进行增删改查；可上传图片、配置推荐权重、查看数据统计与行为分析。 |

系统通过 **JWT 中的 role 字段** 区分用户与管理员；后台所有接口（/api/admin/**）在 Spring Security 中配置为 `hasRole("ADMIN")`，非管理员即使用户已登录也无法访问。

## 二、功能模块详解

### 2.1 用户端功能

- **首页**：轮播图、推荐非遗项目（调用推荐接口）、入口导航。
- **非遗项目**：分页列表，支持按区县、遗产级别、项目类型筛选和关键词搜索；点击进入详情，详情页会记录一次“浏览项目”行为（用于推荐）。
- **非遗资讯**：列表与详情，支持区县筛选与关键词搜索。
- **传承人**：传承人列表与详情，支持区县与关键词。
- **传承故事**：故事列表与详情。
- **非遗活动**：活动列表与详情；详情页显示当前用户是否已报名、报名人数；登录用户可点击“报名”或“取消报名”，报名时写入 activity_registration 并记录一次“报名活动”行为。
- **收藏夹**：展示当前用户收藏的非遗项目列表，可取消收藏。
- **AI 咨询**：用户输入问题，后端调用外部 AI 接口（如 DeepSeek）返回回答。
- **个人中心**：查看与修改昵称、头像、密码等；依赖 /api/user/me 与 PUT /api/user/me。

### 2.2 管理员端功能

- **后台首页（Dashboard）**：展示用户数、非遗项目数等概览统计，以及行为相关图表数据（如各区县浏览量、各类型浏览量、浏览趋势）。
- **非遗项目管理**：列表（支持条件查询）、新增、编辑、删除非遗项目。
- **传承人管理**：传承人 CRUD。
- **非遗资讯管理**：资讯 CRUD。
- **非遗活动管理**：活动 CRUD。
- **传承故事管理**：传承故事 CRUD。
- **用户管理**：用户列表与 CRUD。
- **推荐分析**：查看各类型推荐权重配置（基础权重、浏览权重、收藏权重），修改后保存；查看按类型的浏览量与收藏量统计（analytics）。
- **图片上传**：上传接口 /api/upload，用于头像、项目图片、资讯封面等，上传文件保存在服务端目录（如 src/main/resources/pictures），通过 /pictures/** 静态资源路径访问。

### 2.3 推荐功能（算法简述）

- 数据来源：用户行为表（浏览项目）、收藏表、推荐权重表（按非遗类型配置 baseWeight、viewWeight、favoriteWeight）。
- 流程：读取权重表初始化各类型得分；根据当前用户最近浏览与全部收藏，按项目类型累加浏览增量与收藏增量；按类型得分排序，从得分最高的三种类型中按名额（如 3、2、1）选取项目，过滤已看/已收藏后随机打乱并返回；不足时用全局热门项目兜底。
- 接口：用户端 GET /api/projects/recommend；后台 GET/POST /api/admin/recommend/weights、GET /api/admin/recommend/analytics。

## 三、技术栈及在项目中的具体作用

### 3.1 前端

| 技术        | 版本/说明 | 在项目中的具体作用 |
|-------------|-----------|---------------------|
| **Vue 3**  | Composition API、`<script setup>` | 页面以单文件组件（.vue）组织，如 Home.vue、Projects.vue、各 admin 页面；数据与视图绑定，事件处理（如 @click、@submit）。 |
| **Vue Router** | 前端路由 | 定义路径与组件对应关系（如 / → Home，/projects → Projects，/admin → 后台布局与子路由）；支持路由守卫（如未登录跳转登录页）。 |

| 技术                | 在项目中的具体作用 |
|---------------------|--------------------|
| **Spring Boot 3**   | 整合 Web、Security、JPA 等；DemoApplication 启动类；application.properties/yml 配置端口、数据源、JWT 等。 |
| **Spring Web**     | 提供 @RestController、@RequestMapping、@GetMapping/@PostMapping 等，将 HTTP 请求映射到 Controller 方法，返回 JSON（ResponseEntity 或 Map）。 |
| **Spring Security**| 配置请求级权限：部分路径放行（登录注册、部分 GET），/api/admin/** 需 ADMIN，其余 /api/** 需认证；使用 JWT，无 Session；JwtAuthFilter 在 UsernamePasswordAuthenticationFilter 前解析 Token 并设置 Security 上下文。 |

# 第三部分 Spring Boot 后端流程（详细）

## 六、应用启动过程（简要）

1. 运行 `DemoApplication` 的 main 方法，Spring Boot 启动。
2. 加载 application.properties（数据源、JPA、jwt 等）。
3. 扫描并注册 Bean：各 Controller、Service、Repository、Security 配置、JwtUtil、JwtAuthFilter、WebMvcConfig、InitAdminRunner 等。
4. 执行 `InitAdminRunner.run`：若不存在 role=ADMIN 且 username=admin 的用户，则创建默认管理员 admin/admin123。
5. Spring Security 构建过滤链；内嵌 Tomcat 监听 8080，等待 HTTP 请求。

## 七、单次 HTTP 请求在后端的完整路径

### 7.1 总体顺序

1. **请求进入**  
   浏览器或前端代理发起的请求到达 8080 端口（例如 GET /api/projects?page=0&size=10）。

2. **CORS**  
   若为跨域预检或跨域请求，Spring MVC 的 CORS 配置（WebMvcConfig 中）会添加允许的 Origin、Method、Header 等，使浏览器允许该响应。

3. **Security 过滤链**  
   - 首先经过 `JwtAuthFilter`（在 UsernamePasswordAuthenticationFilter 之前）。
   - `JwtAuthFilter.doFilterInternal`：从请求头读取 `Authorization: Bearer <token>`；若有 token 且 `JwtUtil.validateToken(token)` 为 true，则解析出 userId、role，构造 `UsernamePasswordAuthenticationToken(userId, null, [ROLE_xxx])` 并放入 `SecurityContextHolder.getContext().setAuthentication(...)`；若无或无效则不设置。
   - 然后进入 `FilterSecurityInterceptor`（Spring Security 默认）：根据 SecurityConfig 中的 `authorizeHttpRequests` 规则判断当前请求是否允许访问：
     - 若匹配到 permitAll（如 /api/auth/login、部分 GET /api/projects/** 等），直接放行；
     - 若匹配到 hasRole("ADMIN")（/api/admin/**），则要求 Authentication 非空且含 ROLE_ADMIN，否则 403；
     - 若匹配到 authenticated()（其余 /api/**），则要求 Authentication 非空，否则 401；
   - 通过后继续向下。

4. **DispatcherServlet 与 Controller 匹配**  
   Spring MVC 根据请求路径和 HTTP 方法，匹配到某个 @RestController 的某个方法（如 GET /api/projects → IchProjectController.list）。

5. **参数解析**  
   方法参数由框架注入：@RequestParam、@PathVariable、@RequestBody、@AuthenticationPrincipal 等；@AuthenticationPrincipal Long userId 会从 Security 上下文中取当前主体的 principal（本项目中即 userId）。

6. **Controller 执行**  
   Controller 方法内调用一个或多个 Service 方法，得到业务结果，再封装成 Map 或 ResponseEntity 返回（如 Map.of("success", true, "data", list)）。

7. **序列化为 JSON**  
   Spring 使用 Jackson 将返回值序列化为 JSON 写入响应体，状态码为 200（或 Controller 指定的其他状态码）。

8. **响应返回**  
   HTTP 响应发回给浏览器或代理，前端 axios 收到后在 then 或 async/await 中处理。

### 7.2 Security 配置详解（谁能访问谁）

SecurityConfig 中 `authorizeHttpRequests` 的典型配置含义如下（以你项目为准，此处为常见写法）：

- `requestMatchers("/api/auth/login", "/api/auth/register", "/api/auth/admin/login").permitAll()`  
  这三个路径无需登录即可访问，用于登录与注册。

- `requestMatchers(HttpMethod.GET, "/api/projects/**", "/api/inheritors/**", "/api/news/**", "/api/activities/**", "/api/stories/**", "/api/regions").permitAll()`  
  这些 GET 请求放行，便于未登录用户浏览列表与详情。

- `requestMatchers("/api/admin/**").hasRole("ADMIN")`  
  所有后台接口必须已认证且角色为 ADMIN（JWT 中 role=ADMIN 会变成 ROLE_ADMIN）。

- `requestMatchers("/api/upload/**").authenticated()`  
  上传接口只需登录即可（用户与管理员均可）。

- `requestMatchers("/pictures/**", "/uploads/**").permitAll()`  
  静态资源公开，用于展示上传的图片。

- `requestMatchers("/api/**").authenticated()`  
  其余所有 /api 下接口需要认证（即 JWT 有效且已设置到 Security 上下文）。

- `anyRequest().permitAll()`  
  其他路径（如前端静态页面）放行。

因此：未登录只能访问登录注册和部分 GET 展示接口；登录后可以访问收藏、报名、个人中心、AI 咨询等；管理员才能访问 /api/admin/**。

### 7.3 JWT 的生成与使用

- **生成**：用户或管理员登录成功后，AuthService 调用 `JwtUtil.generateToken(userId, username, role)`，将 userId、username、role 写入 Token 的 payload，并设置过期时间（如 7 天），用配置的 jwt.secret 签名后返回字符串；AuthController 将该 token 放入响应体（如 data.token）返回给前端。
- **前端存储**：前端登录成功后，将 token、userId、role、nickname 等存入 localStorage，并在 Pinia 的 auth store 中保存（setAuth）。
- **携带**：之后每次请求，request.js 的请求拦截器从 localStorage 读取 token，设置 `config.headers.Authorization = 'Bearer ' + token`。
- **校验与解析**：JwtAuthFilter 从请求头取出 Bearer 后的字符串，调用 JwtUtil.validateToken(token)；若通过则用 getUserIdFromToken、getRoleFromToken 解析出用户 id 和角色，并设置到 Security 上下文中，这样 Controller 里 @AuthenticationPrincipal Long userId 即可得到当前用户 id。

## 八、Controller → Service → Repository 各层职责与示例

### 8.1 Controller 层

- **职责**：接收 HTTP 请求、解析参数、调用 Service、将返回结果封装成统一格式（如 { success, data, message }）并设置 HTTP 状态码。
- **不写**：复杂业务逻辑、直接写 SQL。
- **示例**（IchProjectController）：
  - `list(keyword, region, heritageLevel, ichType, page, size)`：调用 `projectService.list(...)` 得到 `Page<IchProject>`，返回 `ResponseEntity.ok(Map.of("success", true, "data", result.getContent(), "total", result.getTotalElements(), "totalPages", result.getTotalPages()))`。
  - `getById(id, userId)`：调用 `projectService.getById(id)` 得到项目；若 userId 非空则调用 `behaviorService.logViewProject(userId, id)` 记录浏览；再返回项目详情。
  - `recommend(userId)`：调用 `recommendationService.recommendProjects(userId, 6)`，返回推荐列表。

### 8.2 Service 层

- **职责**：实现业务逻辑、事务边界、调用多个 Repository 或其它 Service。
- **示例**：
  - **AuthService**：register 时检查用户名/邮箱/手机是否重复，构造 User 并 encode 密码，save 后生成 JWT 返回；login/adminLogin 时按用户名（及角色）查用户，校验密码，生成 JWT 返回。
  - **IchProjectService**：list 方法将 keyword/region/heritageLevel/ichType 转为可空字符串，调用 `projectRepository.searchByConditions(..., PageRequest.of(page, size))` 返回分页结果；getById 使用 `projectRepository.findById(id).orElseThrow(...)`。
  - **RecommendationService**：recommendProjects 中读取权重、浏览与收藏记录，按类型累加得分，排序后按名额从各类型取项目，过滤已看/已收藏，不足则从全局热门补全。
  - **ActivityService**：register 时检查是否已报名（唯一约束），再插入 activity_registration；cancelRegistration 删除对应报名记录。

### 8.3 Repository 层

- **职责**：数据访问，与数据库表一一对应；方法名或 @Query 定义查询，由 Spring Data JPA 生成实现。
- **示例**（IchProjectRepository）：
  - `searchByConditions(keyword, region, heritageLevel, ichType, pageable)`：使用 @Query 写 JPQL，条件中 keyword 为 null 或空时对应条件恒真，其余为等值或 LIKE，返回 `Page<IchProject>`。
  - `findTop20ByIchTypeOrderByApprovalTimeDesc(ichType)`：按类型查前 20 条，按批准时间倒序。
  - `findTop20ByOrderByApprovalTimeDesc()`：全局前 20 条，用于推荐兜底。

# 第四部分 前后端对接（详细）

## 九、开发环境下的请求路径与跨域

- 前端开发服务器：`http://localhost:5173`；后端：`http://localhost:8080`。
- 前端代码中请求写相对路径，例如 `/api/projects`、`/api/auth/login`，不写 8080 的绝对地址。
- 在 **frontend/vite.config.js** 的 `server.proxy` 中配置：
  - `/api` → target: `http://localhost:8080`
  - `/pictures`、`/uploads` → 同样指向 8080
- 浏览器中发起的请求 URL 仍是 `http://localhost:5173/api/projects`，由 Vite 在开发时转发到 8080，因此同源，无跨域问题；生产环境若前后端同域或 Nginx 反向代理同一域名，也可不暴露 8080。

## 十、前端请求封装与登录态

### 10.1 axios 实例（utils/request.js）

- 使用 `axios.create({ baseURL: '', timeout: 15000 })` 创建实例（baseURL 为空表示使用当前域名，即 5173 或部署后的前端域名）。
- **请求拦截器**：每次请求前从 `localStorage.getItem('token')` 读取 token，若存在则 `config.headers.Authorization = 'Bearer ' + token`。
- **响应拦截器**：成功时直接返回 `res.data`（这样业务代码拿到的就是后端返回的 JSON 对象）；失败时若 `err.response.status === 401`，则清除 localStorage 的 token、role、user，并根据当前路径是否在 /admin 下跳转到管理员登录页或用户登录页。

### 10.2 API 封装（api/index.js）

- 将每个后端接口封装成函数，例如：
  - `getProjectList(params)` → `request.get('/api/projects', { params })`
  - `getProjectById(id)` → `request.get('/api/projects/' + id)`
  - `getRecommendProjects()` → `request.get('/api/projects/recommend')`
  - `login(data)` → `request.post('/api/auth/login', data)`
  - `adminLogin(data)` → `request.post('/api/auth/admin/login', data)`
  - `register(data)` → `request.post('/api/auth/register', data)`
  - `getCurrentUser()` → `request.get('/api/user/me')`
  - `registerActivity(id)` → `request.post('/api/activities/' + id + '/register')`
  - `adminGetRecommendWeights()`、`adminSaveRecommendWeights(data)` 等。
- 页面中只 import 这些函数并传入参数，不直接写 URL 和 axios，便于维护和统一带 Token。

### 10.3 登录流程（前端）

- 用户打开 Login.vue，输入账号密码，提交时调用 `login(form)`（即 POST /api/auth/login）。
- 后端 AuthController 调 AuthService.login，校验通过后返回 `{ success: true, data: { token, userId, username, nickname, avatar, role } }`。
- 前端在 then 中若 res.success 且 res.data，则调用 `auth.setAuth(res.data)`：将 token、role、user 写入 Pinia 并同步到 localStorage，然后 `router.push(route.query.redirect || '/')` 跳转首页。
- 之后任意页面发请求时，request 拦截器都会自动带上该 token；若后端返回 401，响应拦截器会清空登录态并跳转登录页。

### 10.4 后端 CORS（WebMvcConfig）

- `addCorsMappings` 中对 `/api/**` 配置：`allowedOriginPatterns("*")`、`allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")`、`allowedHeaders("*")`、`allowCredentials(true)`。
- 这样当前端域名（如 5173 或部署后的域名）请求 8080 时，响应头中会带有允许的 Origin 等，浏览器允许前端 JavaScript 读取响应。

# 第五部分 完整请求链路示例（多条）

## 十一、示例一：用户登录

1. 用户在 Login.vue 输入账号密码，点击登录，触发 `login(form)`。
2. request.js 请求拦截器为请求添加 Authorization（若已有 token）；本次为登录请求，通常尚无 token。
3. 请求发送到 POST /api/auth/login，body 为 { username, password }。
4. Vite 将请求代理到 http://localhost:8080/api/auth/login。
5. 后端 Security：/api/auth/login 为 permitAll，放行。
6. AuthController.login 接收 @RequestBody LoginRequest，调用 AuthService.login(req)。
7. AuthService 通过 UserRepository.findByUsername 查用户，校验角色为 USER，再 PasswordEncoder.matches 校验密码；通过后 JwtUtil.generateToken 生成 token，封装 AuthResponse 返回。
8. Controller 返回 ResponseEntity.ok(Map.of("success", true, "data", resp))，resp 中含 token、userId、username、nickname、avatar、role。
9. 前端收到 res，auth.setAuth(res.data)，跳转首页；之后所有请求头都会带该 token。

## 十二、示例二：非遗项目列表（GET 列表）

1. 用户打开 /projects，Projects.vue 的 onMounted 调用 load(0)。
2. load 内调用 getProjectList({ keyword, region, heritageLevel, ichType, page: 0, size: 12 })。
3. api/index.js 中 getProjectList 即 request.get('/api/projects', { params })；request 拦截器加上 Token（若已登录）。
4. 请求 GET /api/projects?page=0&size=12&...，被代理到 8080。
5. Security：GET /api/projects/** 为 permitAll，放行。
6. IchProjectController.list 接收 @RequestParam 的 keyword、region、heritageLevel、ichType、page、size，调用 projectService.list(...)。
7. IchProjectService.list 调用 projectRepository.searchByConditions(..., PageRequest.of(page, size))，得到 Page<IchProject>。
8. IchProjectRepository 的 @Query 生成 SQL 查询 ich_project 表，结果映射为 List<IchProject> 并带分页信息返回。
9. Controller 将 result.getContent()、getTotalElements()、getTotalPages() 封装成 Map 返回。
10. 前端 res 得到 { success, data, total, totalPages }，将 res.data 赋给 list，template 中 v-for 渲染卡片列表。

## 十三、示例三：首页推荐（GET /api/projects/recommend）

1. 首页 Home.vue 加载时调用 getRecommendProjects()，即 request.get('/api/projects/recommend')；请求头带 Token（若已登录）。
2. GET /api/projects/** 在 Security 中 permitAll，但若带 Token，JwtAuthFilter 仍会解析并设置 userId 到上下文。
3. IchProjectController.recommend 方法参数为 @AuthenticationPrincipal Long userId（未登录则为 null），调用 recommendationService.recommendProjects(userId, 6)。
4. RecommendationService 内：loadWeights() 从 recommendation_weight 表读配置；若 userId 非空则查 recentProjectViews、favorites；按类型累加得分并排序；按 3、2、1 名额从各类型取项目，过滤已看/已收藏，不足则全局热门补全；返回 List<IchProject>。
5. Controller 返回 Map.of("success", true, "data", list)。
6. 前端将 list 展示在首页推荐区域。

## 十四、示例四：活动报名（需登录）

1. 用户在活动详情页点击“报名”，前端调用 registerActivity(activityId)，即 request.post('/api/activities/' + activityId + '/register')；请求头带 Token。
2. Security：/api/activities/{id}/register 的 POST 未单独配置，落在 requestMatchers("/api/**").authenticated()，因此必须已认证；JwtAuthFilter 已从 Token 设置 userId。
3. ActivityController.register 方法参数为 @PathVariable Long id、@AuthenticationPrincipal Long userId；若 userId 为 null 则返回 401；否则调用 activityService.register(userId, id)，再 behaviorService.logRegisterActivity(userId, id)。
4. ActivityService.register 中检查是否已报名（避免重复），再 activityRegistrationRepository.save 新记录（唯一约束 user_id+activity_id 保证唯一）。
5. Controller 返回 200 与 { success: true, message: "报名成功" }；前端可刷新详情页以更新“已报名”状态。

## 十五、示例五：管理员保存推荐权重

1. 管理员在推荐分析页面修改各类型权重后点击保存，前端调用 adminSaveRecommendWeights(rows)，即 request.post('/api/admin/recommend/weights', rows)；请求头带管理员 Token。
2. Security：/api/admin/** 需 hasRole("ADMIN")；JwtAuthFilter 已解析出 role=ADMIN，放行。
3. AdminRecommendationController.saveWeights 接收 @RequestBody List<RecommendationWeight> weights，调用 weightService.saveAll(weights)。
4. RecommendationWeightService.saveAll 中先 repository.deleteAll()，再将传入的 list 中有效类型 trim 后 saveAll 写回数据库。
5. Controller 返回 { success: true, message: "保存成功" }；前端提示保存成功。

# 第七部分 答辩常见问题与回答思路

## 十八、项目与业务类

- **你这个项目解决了什么问题？**  
  丽水非遗资源分散、传播受限；本平台把非遗项目、资讯、传承人、故事、活动集中到线上，支持检索、预约、收藏和基于行为的简单推荐，方便公众了解非遗、管理员统一管理内容。

- **用户和管理员有什么区别？**  
  用户在前台浏览、收藏、预约、用 AI 咨询；管理员通过独立入口登录后台，对项目、资讯、传承人、故事、活动、用户做增删改查，并配置推荐权重、查看统计数据；后台接口在 Spring Security 中限制为 hasRole("ADMIN")。

- **推荐是怎么做的？**  
  根据用户浏览和收藏记录，按非遗项目类型（ich_type）累加权重（基础权重 + 浏览增量 + 收藏增量），类型排序后从高分类型中按名额取项目，过滤已看/已收藏后返回；管理员可在后台配置各类型的基础权重、浏览权重、收藏权重。

## 十九、技术与流程类

- **前后端是怎么连的？**  
  前端用 Vue + Axios 请求相对路径 /api/xxx；开发时 Vite 把 /api 代理到后端 8080，后端用 CORS 允许前端域名；接口返回 JSON，前端按 res.data 渲染；需要登录的接口在请求头里带 Authorization: Bearer <token>，后端用 JWT 校验并识别用户。

- **请求从发起到拿到数据，后端经历了哪些步骤？**  
  请求先进 JwtAuthFilter 解析 Token 并设置用户到 Security 上下文；再经 Security 的权限判断（放行或需登录/需管理员）；然后进入对应 Controller 方法，Controller 调 Service，Service 调 Repository 查库；结果层层返回，Controller 封装成 JSON 通过 HTTP 返回给前端。

- **为什么用 JWT 不用 Session？**  
  本项目采用无状态认证：服务端不存 Session，登录后只签发 JWT，前端存 token 并在每次请求头携带；后端只需验证签名和过期时间即可识别用户，便于水平扩展；Spring Security 配置为 SessionCreationPolicy.STATELESS。

- **持久层用的是什么？**  
  用的是 Spring Data JPA，实体类与表映射，Repository 接口用方法名和 @Query 定义查询，没有使用 MyBatis（若论文中写了 MyBatis，可说明实际实现为 JPA）。

## 二十、代码与实现类

- **Controller 和 Service 分别做什么？**  
  Controller 只负责接收请求、取参数、调 Service、把结果封装成统一 JSON 返回；业务逻辑（如校验、组合查询、推荐算法、事务）都在 Service 里；Controller 不直接操作数据库，由 Service 调 Repository。

- **登录后前端怎么保存登录状态？**  
  登录接口返回的 token、userId、username、role 等存入 Pinia 的 auth store，并同步写到 localStorage；请求拦截器每次从 localStorage 取 token 放到请求头；若后端返回 401，响应拦截器会清空 token 并跳转登录页。

- **默认管理员账号哪来的？**  
  应用启动时执行 InitAdminRunner：若数据库里没有 username=admin 且 role=ADMIN 的用户，就创建一个密码为 admin123 的默认管理员并保存到 sys_user 表；若已存在则不修改，避免覆盖手动改过的密码。
